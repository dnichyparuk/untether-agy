# changelog

This changelog covers changes made in this fork. It adds support for Google's
[Antigravity](https://antigravity.google) CLI (`agy`) as a coding-agent engine,
along with the `/clone`, `/project`, and `/printtimeout` commands and related
fixes. For the history of the upstream project this fork is based on, see the
[littlebearapps/untether changelog](https://github.com/littlebearapps/untether/blob/master/CHANGELOG.md)
(fork diverged after upstream v0.35.3).

## v0.37.1 (2026-07-05)

### fixes

- **fix:** `/clone`, `/project`, `/planmode`, and `/verbose` mutated shared or global state with no permission gate, unlike `/model`/`/reasoning`/`/listen`/`/printtimeout` which all restrict their mutations to group admins (or any sender in a private chat). `/clone` and `/project` write global `[projects.<alias>]` config and (for `/clone`) run an arbitrary `git clone`; `/planmode` controls Claude Code's tool-execution permission mode. Any allowlisted group member â€” not just an admin â€” could previously trigger these. Added the same `require_admin_or_private` gate `/model`/`/reasoning`/`/printtimeout` already use to `handle_clone_command` (`src/untether/telegram/clone.py`) and `handle_project_command` (`src/untether/telegram/new_project.py`). `/planmode` and `/verbose` are `CommandBackend` plugins whose `handle(ctx)` has no access to the Telegram-specific `cfg`/`msg` the gate needs, so their check lives at the shared dispatch chokepoint instead â€” a new `_ADMIN_GATED_COMMAND_IDS` set in `src/untether/telegram/commands/dispatch.py`'s `_dispatch_command`, checked before `backend.handle(ctx)` runs. Found via a cross-command consistency review dimension after `/printtimeout` (v0.37.0) was found and fixed for the same gap. Added `tests/test_command_admin_gate.py` and denial-path tests in `tests/test_clone_command.py`/`tests/test_project_command.py` [#9](https://github.com/dnichyparuk/re-untether/pull/9)
  - Known follow-up (not covered by this fix): `/planmode`'s `/config` menu-button flow invokes the same `PlanModeCommand.handle()` through different plumbing that bypasses the new dispatch-level gate â€” only the text-command path (`/planmode ...`) is covered here.
- **fix:** the thin `build_bot_commands` re-export in `src/untether/telegram/bridge.py` didn't forward the `include_clone`/`include_new_project` keyword arguments its `menu.py` target already accepts, so `bridge.build_bot_commands(runtime, include_clone=True)` raised `TypeError`. Production was unaffected (the startup path `_set_command_menu` calls the `menu.py` function directly), but the stale re-export broke `test_build_bot_commands_includes_clone_when_enabled`. Forwarded both kwargs through the re-export [#9](https://github.com/dnichyparuk/re-untether/pull/9)

## v0.37.0 (2026-07-05)

### changes

- **feat:** new `/printtimeout` command â€” set, show, or clear a per-project override of the Antigravity (`agy`) `--print-timeout` directly from Telegram. Grammar: `/printtimeout` (show effective value + global default), `/printtimeout 30m` / `/printtimeout 1h30m` (set, Go duration syntax, lenient client-side validation), `/printtimeout clear` (revert to global default). New `print_timeout` field on `ProjectSettings`/`ProjectConfig` (`src/untether/settings.py`, `src/untether/config.py`); resolved per-run into the existing `EngineRunOptions` ContextVar (the same channel `model` uses) via a new `AntigravityRunner._resolved_print_timeout()` (`src/untether/runners/run_options.py`, `src/untether/runners/antigravity.py`); injected in the executor's `_run_engine` chokepoint via a new `_apply_project_print_timeout()` helper and a new public `TransportRuntime.project_for_alias()` accessor (`src/untether/telegram/commands/executor.py`, `src/untether/transport_runtime.py`) so both the normal message path and the command-executor `run_one` path pick it up. New module `src/untether/telegram/print_timeout.py` (`handle_print_timeout_command`, `parse_duration`); routed in `loop.py` and added to the static command menu (always available, no section-enabled gate, modeled on `/model`); `printtimeout` reserved as a chat command id. Always writable, but a project whose engine isn't Antigravity still gets the value persisted with a note that it currently only affects Antigravity runs; degrades to a plain hint with no config write when used outside a project topic [#8](https://github.com/dnichyparuk/re-untether/pull/8)

## v0.36.1 (2026-07-05)

### fixes

- **fix:** long-running Antigravity (`agy`) tasks no longer time out at 5 minutes. `agy`'s own `--print-timeout` defaults to `5m0s`, and Untether left `[antigravity] print_timeout` unset â€” so any task running longer than 5 minutes was killed by `agy` and surfaced to Telegram as a "timeout". Untether now defaults `print_timeout` to `15m` (passing `--print-timeout 15m`), still overridable via `[antigravity] print_timeout` with Go duration syntax. Changed the dataclass default and `build_runner` fallback in `src/untether/runners/antigravity.py`; updated `docs/reference/runners/antigravity/runner.md` and `docs/reference/config.md`; added `test_build_runner_print_timeout_override` and default assertions in `tests/test_antigravity_runner.py` [#7](https://github.com/dnichyparuk/re-untether/issues/7)
  - Raising the default surfaced a side effect: antigravity emits no interim `ActionEvent`s, so the bridge's event-silence stall monitor (`runner_bridge.py:_stall_monitor`) would fall to its 5-minute "normal" threshold and warn on every legitimate 5-15 minute run. Added an antigravity-specific 15-minute stall threshold matching the new `print_timeout` default so healthy runs complete before a stall warning fires; corrected `runner.md`'s prior (incorrect) claim that the stall watchdog is inert for this engine.
  - Added `--print-timeout` to `_RESERVED_FLAGS` in `src/untether/runners/antigravity.py` so `[antigravity] extra_args` can no longer append a duplicate `--print-timeout`, which would silently defeat the configured default; added `test_build_runner_rejects_duplicate_print_timeout_flag`.

## v0.36.0 (2026-07-04)

### changes

- **feat:** new /project command â€” register a brand-new local project directly from Telegram without cloning any repo. Grammar: /project <name>. Sanitizes the name to an alias, refuses (naming the existing path) if the alias already exists (no dedupe, unlike /clone) and refuses a pre-existing non-empty destination directory (mirroring /clone's conflict guard), creates an empty destination dir under [new_project] root, writes a [projects.<alias>] entry, and â€” in a forum-enabled group in scope â€” creates and binds a topic (degrades to register-only otherwise), reusing the shared create_and_bind_topic helper extracted from /clone. New [new_project] config section (enabled, root, default_engine), hot-reloadable; project reserved as a chat command id. New module src/untether/telegram/new_project.py (resolve_project_path, handle_project_command) plus sanitize_alias factored out of clone.derive_alias [#6](https://github.com/dnichyparuk/re-untether/pull/6)
- **feat:** new Antigravity (`agy`) CLI runner â€” control Google's Antigravity agent from Telegram alongside the existing engines. Non-interactive, single-shot: Untether invokes `agy` with `--output-format json` and parses one structured result envelope (`conversation_id`, `status`, `response`, `usage`) rather than a live JSONL stream, so the runner ships at a reduced capability tier (no live progress / no interactive permission control â€” those remain Claude-only). Cross-environment `/continue` is supported via `agy`'s machine-global `--continue`. New `src/untether/telegram/../runners/antigravity.py`, registered as an engine backend entry point; reference docs under `docs/reference/runners/antigravity/` (runner spec, result-envelope cheatsheet, event mapping) plus a usage/configuration guide. Robustness pass hardens error handling and result parsing for missing/partial envelopes [#3](https://github.com/dnichyparuk/re-untether/pull/3)
- **feat:** new `/clone` command â€” clone a GitHub repo directly from Telegram and auto-register it as a project, no terminal round-trip required. Grammar: `/clone <repo-url> [--dir <path>] [@<branch>]`, accepting both `https://github.com/OWNER/REPO[.git]` and scp-style `git@github.com:OWNER/REPO[.git]` URLs. Runs a native `git clone` (shallow by default, `--depth 1 --single-branch`) with host git credentials used as-is (no token injection in v1; `GIT_TERMINAL_PROMPT=0` + batch-mode `GIT_SSH_COMMAND` make an auth-required clone fail fast instead of hanging), derives a project alias from the repo name, and writes a `[projects.<alias>]` entry â€” the same shape `untether init` produces. In a forum-enabled group in scope, also creates a topic bound to the new project; in private chats or non-forum/topics-disabled groups it degrades to register-only with a hint at the equivalent `/topic` command. New `[clone]` config section (`enabled`, `root`, `allowed_hosts`, `default_engine`, `depth`) â€” see the [config reference](docs/reference/config.md#clone). New module `src/untether/telegram/clone.py` (`RepoRef`, `parse_repo_url`, `derive_alias`, `resolve_destination`, `host_is_allowed`, `run_git_clone`, `register_project`, `handle_clone_command`), routed as a top-level command in `loop.py` so it works outside the topics-enabled guard [#3](https://github.com/dnichyparuk/re-untether/pull/3)
